Part 1 · Conceptual Foundations

The Document Web vs. The Semantic Web

The Web we know

  • Pages written for humans
  • Search engines index words
  • Links connect documents
  • No shared understanding of meaning

The Semantic Web

  • Data readable by machines
  • Links connect things (not pages)
  • A global knowledge graph
  • Shared, unambiguous meaning

Note

Tim Berners-Lee coined the term in 2001. The goal: make the Web a universal medium for data exchange, not just document retrieval.

The Core Idea: Triples

Everything in the Semantic Web is a triple: Subject — Predicate — Object

<Photo_001>   dc:creator             "Julien Rabaud"
<Photo_001>   dc:subject             <geonames:2988507>     # Paris
<Photo_001>   cidoc:P138_represents  <wd:Q5783414>          # the building

Tip

Triples form a graph — each resource can be both subject and object.
This creates a web of interconnected knowledge.

URIs — Naming Things on the Web

A URI is a unique, global, persistent identifier for any thing — a person, a place, a concept.

https://sws.geonames.org/2988507/          →  Paris
https://www.wikidata.org/entity/Q937       →  Marie Curie
http://vocab.getty.edu/page/aat/300015646  →  photographs

Unlike a URL (which locates a web page), a URI names a resource — in any language, forever.

Why URIs?

  • Globally unique
  • Persistent
  • Language-independent
  • Dereferenceable

Tip

When you put a GeoNames URI in a Tropy field, you’re linking your data to a global knowledge graph — not just typing a string.

RDF — Expressing Triples

RDF (Resource Description Framework) is the W3C standard for expressing triples.

Turtle — human-readable:

@prefix dc: <http://purl.org/dc/elements/1.1/> .

<https://myarchive.org/photo/001>
  dc:title   "Cloister of Santo Domingo" ;
  dc:creator "Julien Rabaud" ;
  dc:date    "2024-09-15" .

JSON-LD — used by Tropy:

{
  "@context": {
    "dc": "http://purl.org/dc/elements/1.1/"
  },
  "@id": "https://myarchive.org/photo/001",
  "dc:title": "Cloister of Santo Domingo",
  "dc:creator": "Julien Rabaud"
}

Note

Tropy stores all metadata as JSON-LD internally — your descriptions are already Linked Data.

Metadata Schemas

How do we describe what we see?

Dublin Core — 15 Universal Elements

The simplest and most widespread metadata standard. Works for any resource type.

Property Usage
dc:title Title
dc:creator Photographer / author
dc:date Date of creation
dc:subject Topic / theme
dc:description Free-text note
dc:type photograph, drawing…
dc:rights License / copyright
Property Usage
dc:format JPEG, TIFF…
dc:identifier Shelfmark / ID
dc:source Archive, repository
dc:language Language of visible text
dc:relation Related items
dc:coverage Place or period
dc:publisher Institution

🔗 dublincore.org · Namespace: http://purl.org/dc/elements/1.1/

Dublin Core Terms (DCTERMS)

dcterms: provides richer, typed versions — plus extra properties.

# Basic DC — untyped string
dc:date "2024-09-15"

# DCTERMS — typed, linkable
dcterms:created    "2024-09-15"^^xsd:date
dcterms:spatial    <geonames:3117735>
dcterms:license    <https://creativecommons.org/licenses/by/4.0/>
dcterms:isPartOf   <myArchive:collection_42>

Key extras:

  • dcterms:spatial — place URI
  • dcterms:temporal — time period
  • dcterms:license — explicit license URI
  • dcterms:isPartOf — collection hierarchy
  • dcterms:abstract — extended description

Both dc: and dcterms: are available as Tropy template vocabularies.

CIDOC-CRM — Cultural Heritage Standard

The Conceptual Reference Model of the CIDOC — international standard for cultural heritage data (Europeana, museums, major archives).

Unlike Dublin Core (which describes documents), CIDOC-CRM models events, actors, places, and objects.

Note

Key insight: In CIDOC-CRM, everything important happens.
A photograph wasn’t just taken — it was taken by someone, somewhere, at a moment, of something.

Core classes:

Class Meaning
E22 Human-Made Object
E31 Document / Photograph
E39 Actor (person/group)
E52 Time-Span
E53 Place
E65 Creation event

🔗 cidoc-crm.org

CIDOC-CRM in Practice

<Photo_001>     rdf:type               cidoc:E31_Document
<Photo_001>     cidoc:P138_represents  <wd:Q5783414>           # the depicted building

<Creation_001>  rdf:type               cidoc:E65_Creation
<Creation_001>  cidoc:P14_carried_out_by  <orcid:0000-0002-6604-9777>
<Creation_001>  cidoc:P4_has_time-span    <2024-09-15>
<Creation_001>  cidoc:P7_took_place_at    <geonames:3117735>

Tip

In Tropy, you can import the CIDOC-CRM vocabulary and build event-oriented templates.
The Tropy Correspondence template is already structured around events.

Controlled Vocabularies

Consistent, linked values for metadata fields

Why Controlled Vocabularies?

The free-text problem:

  • “Paris” / “paris” / “Paris, France” → 3 different strings
  • “19th century” / “1800s” / “XIX c.” → no shared meaning
  • No multilingual support
  • No machine-readable links

The URI solution:

dcterms:spatial <geonames:2988507>

→ Always Paris, in any language
→ With coordinates and hierarchy
→ Linked to millions of other places

Note

Controlled vocabularies = shared dictionaries.
When we all use the same URI for Paris, our datasets become interoperable.

GeoNames

12+ million geographic names, all with stable URIs.

https://sws.geonames.org/2988507/  →  Paris, France
https://sws.geonames.org/3117735/  →  Salamanca, Spain
https://sws.geonames.org/6440564/  →  Anglet, France

Each entry: names in 20+ languages · coordinates · administrative hierarchy · feature type.

Tip

In Tropy: use GeoNames URIs in the dcterms:spatial field of your template.
🔗 geonames.org/search.html

Pactols & Openthéso

PACTOLS — multilingual thesaurus for Archaeology, Classical and Oriental Studies (Frantiq network).

Covers: periods · object types · materials · ancient places · historical figures

Especially relevant for CHORAL research (Romance cultures, Mediterranean heritage).

🔗 opentheso.frantiq.fr

Huma-Num Openthéso — platform hosting many discipline-specific vocabularies.

Terms use ARK identifiers and SKOS:

  • skos:prefLabel — preferred label
  • skos:altLabel — synonyms
  • skos:broader / skos:narrower
  • skos:exactMatch — links to other thesauri

🔗 opentheso.huma-num.fr

Part 2 · IIIF

International Image Interoperability Framework

What is IIIF?

A set of open standards for rich access to digitized images from cultural heritage institutions.

Developed since 2011 by libraries, archives, and museums (BnF, British Library, Harvard, Europeana…).

Note

The core promise:
Any IIIF-compliant viewer can display any IIIF-compliant image — regardless of where it is hosted.

IIIF enables:

  • 🔍 Deep zoom on high-res images
  • 📐 Request regions, sizes, rotations
  • 📋 Structured collections (Manifests)
  • 🔗 Embedding across institutions
  • 💬 Shared annotation layers
  • 🤝 Multi-institutional exhibitions

🔗 iiif.io

The Two Core APIs

Image API

How images are served.

{server}/{identifier}
  /{region}/{size}/{rotation}
  /{quality}.{format}

# 800px-wide version:
https://example.org/iiif/photo001
  /full/800,/0/default.jpg

→ Deep zoom, cropping, resizing — server-side

Presentation API

How collections are described.

A Manifest is a JSON-LD document:

  • Which images belong together
  • Their order and structure
  • Labels and metadata
  • Annotations
{
  "type": "Manifest",
  "label": { "en": ["My Photo Series"] },
  "items": [...]
}

The Manifest Structure

Manifest
  └── Canvas  (= one "view" or "page")
       ├── Annotation Page → Annotation [painting]   →  the image
       └── Annotation Page → Annotation [commenting] →  your notes

Tip

Think of a Manifest as a virtual object:

  • A manuscript → 1 Manifest, 200 Canvases
  • A photo series → 1 Manifest, 50 Canvases
  • An altarpiece → 1 Manifest, front + back + details

When you import a Manifest in Arvest, you get the full structure — images, metadata, and any existing annotations.

IIIF Sources: Europeana

Europeana aggregates heritage from 3,000+ institutions, with IIIF access to millions of items.

Finding IIIF content:

  • Browse europeana.eu → look for the IIIF icon
  • Manifest URL pattern:
https://iiif.europeana.eu/presentation/
  {collection}/{recordId}/manifest

Note

For CHORAL researchers:
Search for heritage from Spain, Portugal, France, Italy, Romania — manuscripts, photographs, maps, objects.

🔗 europeana.eu · API: pro.europeana.eu/apis

Part 3 · Tools

Tropy

From archival photos to structured, linked research data

What is Tropy?

A free, open-source desktop app for organizing and describing research photographs.

Not a generic photo manager — built for historians, art historians, and humanists.

Core workflow:

  1. Import photos from your archival sessions
  2. Group related photos into items
  3. Describe with rich metadata
  4. Annotate regions of interest
  5. Export as JSON-LD, CSV, or to Omeka S

About Tropy:

Created by RRCHNM (George Mason U.) & C²DH (U. Luxembourg). First release: 2017.

🔗 tropy.org
📚 docs.tropy.org
💬 community.tropy.org

The Tropy Interface

4 main panels:

  • Project (left) — lists, tags, saved searches
  • Item grid / table (center) — browse all items
  • Metadata panel (right) — describe the selected item
  • Viewer — view and annotate photos

Tip

One Tropy project per research trip or archive collection.

Two project modes:

Mode Description
Standard Copies files → portable
Advanced Links to originals → lighter

For archival work: Standard is safer.

The Three Levels of Description

📁 Item

Groups logically related photos.

Example: a document photographed recto/verso = 1 item, 2 photos.

Main unit of metadata: title, date, archive reference.

🖼️ Photo

One image file.

Metadata: filename, dimensions, date taken.

You rarely describe photos individually — the item is the primary unit.

🔍 Selection

A cropped region of a photo.

Examples: a seal, a signature, an inscription.

Has its own title, notes, tags — linked to pixel coordinates.

Templates & Schemas

A template defines which metadata fields appear for an item.

Built-in templates:

  • Tropy Generic — Dublin Core-based
  • Tropy Correspondence — for letters
  • Tropy Audio / Video
  • Tropy Photo — photo-level metadata

Importing external vocabularies:

Edit → Preferences → Vocabularies → Import

→ Any RDF/OWL schema (JSON-LD or Turtle)
→ Then use those properties in your templates

Note

You can build a CIDOC-CRM-aligned template by importing the CIDOC-CRM vocabulary and selecting relevant properties (P138_represents, P7_took_place_at…).

Plugins

Import:

  • CSV Import — from a spreadsheet
  • Omeka S Import — pull items from Omeka S
  • IIIF Import — from a IIIF Manifest URL

Export:

  • Omeka S Export — push to Omeka S
  • IIIF Export — generate Manifests
  • CSV Export — spreadsheet

Arvest

Annotate, collaborate, expose IIIF collections

What is Arvest?

A web platform for working with IIIF image collections.

Key features:

  • Import local images or IIIF Manifests
  • Annotate with W3C Web Annotation standard
  • Collaborative workspaces with roles
  • REST API for data access
  • Export for machine learning pipelines

Note

No installation needed — works in your browser.

🔗 arvest.app
📚 docs.arvest.app

Importing Images

From local files:

  1. Create a workspace
  2. Import → Upload files
  3. Arvest generates a IIIF Manifest automatically

Supported: JPEG, PNG, TIFF, WebP

From a IIIF Manifest URL:

  1. Find a Manifest (Europeana, Nakala, Omeka S…)
  2. Import → IIIF Manifest URL
  3. Paste → all canvases imported
https://gallica.bnf.fr/iiif/ark:/12148/
  btv1b8452439r/manifest.json

Annotating in Arvest

Uses the W3C Web Annotation standard.

Annotation types:

  • Region — box or polygon on the image
  • Point — mark a specific location
  • Full-canvas — note on the whole image

Each annotation: body (text, tag, URI) · motivation · creator + date.

Uses in humanities research:

  • Identify depicted persons, places, objects
  • Transcribe inscriptions or captions
  • Tag iconographic themes
  • Link regions to controlled vocabulary URIs
  • Compare motifs across a corpus

Collaborating in Arvest

Shared workspaces:

  • Invite collaborators by email
  • Roles: viewer · annotator · editor · admin
  • All annotations visible to the team
  • Comment threads · activity log

Workflow:

  1. Import corpus + set guidelines
  2. Collaborators annotate independently
  3. Review — conflicts flagged
  4. Consensus — annotations finalized
  5. Export for publication or ML

Tip

For CHORAL: invite partners across institutions to annotate the same corpus — across national borders.

Exposing Data: API & ML

REST API:

GET /api/v1/workspaces/{id}/manifests
GET /api/v1/workspaces/{id}/annotations
GET /api/v1/manifests/{id}/canvas/{n}

→ Your annotated corpus becomes a queryable dataset

For Machine Learning:

Export annotated regions as:

  • COCO — object detection
  • CSV / JSON — text classification
  • IIIF + W3C — ML pipelines

→ Annotate once, reuse everywhere.
