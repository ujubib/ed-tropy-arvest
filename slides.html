Part 1

The Semantic Web & Linked Data

The Web We Know vs. The Semantic Web

The Document Web

  • Pages written for humans
  • Search engines index words
  • No shared understanding of meaning
  • Links connect documents, not data

The Semantic Web

  • Data written for machines (and humans)
  • Machines can infer meaning
  • A global, interconnected knowledge graph
  • Links connect things, not just documents

Tim Berners-Lee (inventor of the Web) coined the term in 2001.
The goal: make the Web a universal medium for data exchange.

The Core Idea: Triples

Everything in the Semantic Web is expressed as a triple:

Subject
What are we talking about?

Predicate
What property?

Object
What value?

<Photo_001> <dc:creator> “Julien Rabaud”

<Photo_001> <dc:subject> <geonames:2988507> (Paris)

<Photo_001> <cidoc:P138_represents> <wd:Q90>

💡 Triples form a graph — each resource can be both subject and object, creating a web of interconnected knowledge.

URIs — Uniform Resource Identifiers

A URI is a unique, global identifier for any thing — a person, a place, a concept, a document.

Unlike a URL (which locates a web page), a URI names a resource.

https://sws.geonames.org/2988507/   # Paris
https://www.wikidata.org/entity/Q937  # Marie Curie
http://vocab.getty.edu/page/aat/300015646  # photographs
https://pactols.frantiq.fr/opentheso/?idt=TH_1&idc=7

Why URIs?

  • Globally unique
  • Persistent (stable over time)
  • Dereferenceable (can be looked up)
  • Language-independent

In practice

When you assign a GeoNames URI to a location field in Tropy, you’re not just typing a string — you’re linking your data to a global knowledge graph.

RDF — The Language of Triples

RDF (Resource Description Framework) is the W3C standard for expressing triples.

Turtle syntax (human-readable):

@prefix dc: <http://purl.org/dc/elements/1.1/> .
@prefix geo: <https://sws.geonames.org/> .

<https://myarchive.org/photo/001>
  dc:title "Cloister of Santo Domingo" ;
  dc:creator "Julien Rabaud" ;
  dc:date "2024-09-15" ;
  dc:spatial geo:3117735 .

JSON-LD (used by Tropy!):

{
  "@context": {
    "dc": "http://purl.org/dc/elements/1.1/"
  },
  "@id": "https://myarchive.org/photo/001",
  "dc:title": "Cloister of Santo Domingo",
  "dc:creator": "Julien Rabaud"
}

Tropy stores metadata as JSON-LD internally — your descriptions are already Linked Data.

Metadata Schemas: Properties

How do we describe what we see?

Dublin Core

The simplest and most widespread metadata standard. 15 core elements, usable for any resource type.

Property Usage
dc:title Title of the image
dc:creator Photographer / author
dc:date Date of creation
dc:subject Subject / topic
dc:description Free text description
dc:type Type (photograph, drawing…)
Property Usage
dc:format File format (JPEG, TIFF…)
dc:identifier Unique ID / shelfmark
dc:source Archive, repository
dc:language Language of text visible
dc:rights Licence / copyright
dc:relation Related resources

🔗 dublincore.org · Namespace: http://purl.org/dc/elements/1.1/

Dublin Core Extended (DCTERMS)

Dublin Core Terms (dcterms:) provides richer, typed versions of the 15 elements plus additional properties.

# Basic DC
dc:date "2024-09-15"

# Richer DCTERMS
dcterms:created "2024-09-15"^^xsd:date
dcterms:spatial <geonames:3117735>
dcterms:license <https://creativecommons.org/licenses/by/4.0/>
dcterms:isPartOf <myArchive:collection_42>

Key extras in DCTERMS:

  • dcterms:spatial — geographic coverage
  • dcterms:temporal — time period covered
  • dcterms:license — explicit license URI
  • dcterms:isPartOf / hasPart — collection hierarchy
  • dcterms:abstract — extended description

🔗 Namespace: http://purl.org/dc/terms/ · Both dc: and dcterms: are available as Tropy templates

CIDOC-CRM

The Conceptual Reference Model of the CIDOC (International Committee for Documentation) — the international standard for cultural heritage data.

Unlike Dublin Core (which describes documents), CIDOC-CRM describes events, actors, places, and objects and their relationships.

Key insight: In CIDOC-CRM, everything important happens.
A painting wasn’t just created — it was created by a person, at a place, during a period, using materials, and was later moved, acquired, exhibited…

Core classes:

  • E1 Thing
  • E18 Physical Thing
  • E22 Human-Made Object
  • E31 Document / Photograph
  • E53 Place
  • E52 Time-Span
  • E39 Actor
  • E65 Creation

🔗 cidoc-crm.org · Used in Europeana, museum databases, and Nakala

CIDOC-CRM in Practice

<Photo_001> rdf:type cidoc:E31_Document

<Photo_001> cidoc:P138_represents <wd:Q5783414> (the depicted building)

<Creation_event> rdf:type cidoc:E65_Creation

<Creation_event> cidoc:P14_carried_out_by <orcid:0000-0002-6604-9777>

<Creation_event> cidoc:P4_has_time-span <2024-09-15>

<Creation_event> cidoc:P7_took_place_at <geonames:3117735>

Note

For Tropy users: The Tropy Generic template approximates Dublin Core. The Tropy Correspondence template is more event-oriented. You can import a CIDOC-CRM-aligned template via the JSON-LD schema import feature.

Controlled Vocabularies: Values

Consistent, linked values for your metadata fields

Why Controlled Vocabularies?

The problem with free text:

  • “Paris” / “paris” / “PARIS” / “Paris, France” — all different strings
  • “19th century” / “1800s” / “XIX century”
  • No way to query across datasets
  • No multilingual support

The solution — URIs as values:

dc:spatial <geonames:2988507>
# → always Paris, in any language
# → machine-readable coordinates
# → links to 11M other GeoNames entries

Controlled vocabularies = shared dictionaries
When we all use the same URI for Paris, our datasets become interoperable — they can be merged, queried, and compared.

GeoNames

GeoNames is a geographical database covering all countries, with over 12 million place names, all with stable URIs.

https://sws.geonames.org/2988507/   → Paris
https://sws.geonames.org/3117735/   → Salamanca
https://sws.geonames.org/6440564/   → Anglet

Each entry includes: - Names in multiple languages - Coordinates (latitude/longitude) - Administrative hierarchy (city → region → country) - Feature type (mountain, river, church…)

In Tropy:

Use GeoNames URIs in the dc:spatial or dcterms:spatial field.

This makes your geographic metadata globally interoperable!

🔗 geonames.org · Search: geonames.org/search.html

Pactols

PACTOLS is a multilingual thesaurus for Archaeology, Classical Studies and Oriental Studies, maintained by the Frantiq network (French research infrastructure).

Covers subjects like: - Archaeological periods and cultures - Object types and materials - Geographic places (ancient and modern) - Historical persons and mythological figures - Techniques and practices

https://pactols.frantiq.fr/opentheso/?idt=TH_1&idc=12345
# → a specific archaeological period

Why Pactols?

Especially relevant for researchers working on:

  • Classical antiquity
  • Medieval heritage
  • Mediterranean archaeology
  • Romance language cultures (CHORAL!)

🔗 frantiq.fr · Thesaurus: opentheso.frantiq.fr

Huma-Num Openthéso

Openthéso is an open-source thesaurus management platform developed by Huma-Num (French national infrastructure for Digital Humanities).

It hosts numerous discipline-specific thesauri:

  • PACTOLS (Archaeology)
  • Thésaurus de la désignation des œuvres architecturales (MHFA)
  • Vocabulaire des arts de la scène
  • … and many others

Each term has a stable URI (ARK identifier) and SKOS representation.

Key feature: SKOS

Terms are expressed in SKOS (Simple Knowledge Organization System):

  • skos:prefLabel (preferred label)
  • skos:altLabel (synonyms)
  • skos:broader / skos:narrower
  • skos:exactMatch (links to other thesauri!)

🔗 opentheso.huma-num.fr · Docs: opentheso.github.io

IIIF — The International Image Interoperability Framework

What is IIIF?

IIIF (pronounced “triple-eye-eff”) is a set of open standards that enable rich access to digitized images from cultural heritage institutions.

Developed since 2011 by a community of libraries, archives, and museums (including BnF, British Library, Harvard, Stanford, Europeana…).

The core promise:
Any IIIF-compliant viewer can display any IIIF-compliant image, regardless of where it’s hosted.

IIIF enables:

  • 🔍 Deep zoom on high-resolution images
  • 📐 Requesting specific regions, sizes, rotations
  • 📋 Structured collections (Manifests)
  • 🔗 Linking and embedding across institutions
  • 💬 Shared annotation layers
  • 🤝 Multi-institutional exhibitions

🔗 iiif.io · Community: iiif.io/community

The Key IIIF APIs

Image API

Standardizes how images are served.

URL pattern:

{server}/{prefix}/{identifier}
  /{region}/{size}/{rotation}
  /{quality}.{format}

Example:

https://example.org/iiif/img01
  /full/800,/0/default.jpg

→ Enables deep zoom, cropping, resizing — all server-side

Presentation API (Manifests)

Standardizes how image collections are described.

A Manifest is a JSON-LD document that says: - What images belong to this object/collection - In what order - With what labels and metadata - With what annotations

{
  "@context": "http://iiif.io/api/presentation/3/context.json",
  "id": "https://example.org/manifest/1",
  "type": "Manifest",
  "label": { "en": ["My Archive"] }
}

The Manifest — Heart of IIIF

Manifest (Collection or single object)
  └── Canvas (= a "page" or "view")
       ├── Annotation Page
       │    └── Annotation (painting role → the image)
       └── Annotation Page
            └── Annotation (commenting role → your notes!)

Think of a Manifest as a virtual object:
A manuscript has 200 pages → 200 Canvases.
A photo series has 50 images → 50 Canvases.
A painted altarpiece has front, back, details → multiple Canvases.

Tip

For Arvest users: When you import a IIIF Manifest in Arvest, you’re importing the entire structure — all canvases, their images, their metadata, and any existing annotations.

Europeana IIIF

Europeana aggregates cultural heritage from 3,000+ institutions across Europe and provides IIIF access to millions of items.

Finding IIIF content: - Browse europeana.eu - Look for the IIIF icon on item pages - Or use the Europeana API with iiif=true

https://iiif.europeana.eu/presentation/
  {collection}/{recordId}/manifest

For CHORAL researchers:

Search Europeana for: - Heritage from Spain, Portugal, France, Italy, Romania - Manuscripts in Romance languages - Historical photographs and maps - Museum objects with provenance data

🔗 europeana.eu · API: pro.europeana.eu/apis

Omeka S & IIIF

Omeka S (used by UPPA) natively generates IIIF Manifests for its items and media.

Any item with an image in your Omeka S instance has a IIIF endpoint:

https://omeka.univ-pau.fr/iiif/3/{itemId}/manifest

And a media endpoint:

https://omeka.univ-pau.fr/iiif/3/media/{mediaId}/canvas/p1

UPPA Omeka S

  • Internal research collections
  • Public heritage exhibitions
  • Metadata aligned with Dublin Core and CIDOC-CRM
  • Auto-generated IIIF Manifests

→ Directly importable in Arvest!

🔗 omeka.org · Module: iiif.io Omeka module

Nakala — Huma-Num Data Repository

Nakala is a research data repository for the Humanities, operated by Huma-Num (CNRS).

It provides: - Stable storage with DOIs for datasets - IIIF image serving for all deposited images - API access to metadata and files - Public and private data modes

Any image deposited in Nakala gets a IIIF Image API endpoint automatically.

For your research:

  1. Deposit photos/scans in Nakala
  2. Get a persistent DOI + IIIF URL
  3. Import the Manifest into Arvest
  4. Annotate and expose for collaborators

→ A complete, FAIR-compliant workflow!

🔗 nakala.fr · Docs: documentation.huma-num.fr/nakala

Tools for Working with Image Corpora

Tropy

From archival photos to structured, linked research data

What is Tropy?

Tropy is a free, open-source desktop application for organizing and describing research photographs.

Think of it as a research photo manager built specifically for historians, art historians, and humanists — not a generic photo app.

The core workflow:

  1. Import photos from your archival sessions
  2. Group related photos into items
  3. Describe with rich metadata (any schema)
  4. Annotate regions of interest
  5. Export as JSON-LD, CSV, or to Omeka S

History & Links

  • Created by Roy Rosenzweig Center for History and New Media (George Mason U.) & C²DH (U. Luxembourg)
  • First release: 2017
  • Current version: 2.x
  • License: AGPL-3.0

🔗 tropy.org
📦 github.com/tropy/tropy
📚 docs.tropy.org

The Tropy Interface

Main areas:

  • Project Panel (left) — lists, tags, saved searches
  • Item Grid/Table (center) — browse all items
  • Metadata Panel (right) — describe the selected item or photo
  • Viewer (bottom/center) — view and annotate photos

Two project modes:

Standard project — copies files to project folder → portable, self-contained

Advanced project — links to original files → lighter, but path-dependent

→ For archival work: Standard is safer

Tip

Good practice: Create one Tropy project per research trip or archive collection.

The Three Levels of Description

Tropy structures data in three nested levels, each with its own metadata:

📁 Item

Groups logically related photos

Example:
A multi-page document photographed front and back → 1 item, 2 photos

Metadata: title, date, archive reference, box/folder number

🖼️ Photo

One image file

Metadata: filename, dimensions, date taken, camera settings

Note: you rarely describe photos individually — the item is the unit of description

🔍 Selection (Detail)

A cropped region of a photo

Example:
A detail of a seal, a signature, an inscription, a decorative motif

Metadata: its own title, notes, tags — linked to coordinates in the image

Templates — Defining Your Metadata Schema

A template in Tropy defines which fields appear in the metadata panel for an item.

Each field is a property from a vocabulary (Dublin Core, CIDOC-CRM, custom, etc.).

Built-in templates:

  • Tropy Generic — Dublin Core-based
  • Tropy Correspondence — letters and correspondence
  • Tropy Audio / Tropy Video
  • Tropy Photo — photo-level metadata

Custom templates:

  • Create your own via Edit → Templates
  • Add/remove/reorder properties
  • Set default values and required fields

Importing external schemas:

Tropy can import vocabulary schemas from the web:

  • Any RDF/OWL vocabulary published as JSON-LD or Turtle
  • Then use those properties in your templates

Edit → Preferences → Vocabularies → Import

Pre-packaged schemas: github.com/tropy/tropy/tree/main/db/vocabularies

Useful Tropy Plugins

Import Plugins

CSL/CSV Import — import records from a spreadsheet

Omeka S Import — pull items from your Omeka S instance into Tropy (metadata + images)

Arvest

Working with IIIF collections: annotate, collaborate, expose

What is Arvest?

Arvest (arvest.app) is a web-based platform for working with IIIF image collections.

It was developed by the IIIF community and designed for digital humanities researchers who want to:

  • Work with images from multiple sources (local, institutional repositories, Europeana…)
  • Annotate images individually or collaboratively
  • Expose their curated collections via APIs
  • Use image collections as input for machine learning pipelines

Key features:

  • IIIF Manifest import (URL or file)
  • Local image upload → auto-IIIF
  • Web annotation (W3C standard)
  • Collaborative workspaces
  • REST API for data access
  • ML-ready data export

🔗 arvest.app · Docs: docs.arvest.app

Importing Images into Arvest

From Local Files

  1. Login to arvest.app
  2. Create a new Workspace
  3. Click Import → Upload files
  4. Select images from your computer
  5. Arvest generates a IIIF Manifest automatically

Supports: JPEG, PNG, TIFF, WebP

From a IIIF Source

  1. Find a IIIF Manifest URL
    • From Europeana, Nakala, Omeka S, BnF Gallica, Wellcome…
  2. Click Import → IIIF Manifest URL
  3. Paste the Manifest URL
  4. Arvest imports all canvases and metadata

Examples:

https://gallica.bnf.fr/iiif/ark:/12148/btv1b8452439r/manifest.json
https://nakala.fr/10.34847/nkl.xxx/manifest

Annotating in Arvest

Arvest uses the W3C Web Annotation standard — the same standard as IIIF annotation layers.

Annotation types:

  • Region annotations — draw a box or polygon on the image
  • Point annotations — mark a specific location
  • Full-canvas annotations — notes for the whole image

Each annotation can have: - A body (text, tag, linked URI) - A motivation (describing, tagging, assessing…) - Creator and date metadata

Practical uses:

  • Identify depicted persons, places, objects
  • Transcribe visible text (inscriptions, captions)
  • Tag iconographic themes
  • Link regions to controlled vocabulary terms
  • Compare details across images in your corpus

Collaborating in Arvest

Shared Workspaces:

  • Invite collaborators via email
  • Assign roles: viewer, annotator, editor, admin
  • See all annotations from all collaborators
  • Comment threads on annotations
  • Activity log

Tip

For CHORAL projects:
Create a shared workspace → invite partners across institutions → annotate the same corpus collaboratively — even across national borders!

Annotation workflows:

  1. Lead researcher imports corpus + sets annotation guidelines
  2. Collaborators annotate independently
  3. Review — annotation conflicts flagged
  4. Consensus — approved annotations marked final
  5. Export for publication or ML

Exposing Data via API

Arvest provides a REST API to expose your curated workspace data:

GET /api/v1/workspaces/{id}/manifests
→ All IIIF Manifests in workspace

GET /api/v1/workspaces/{id}/annotations
→ All annotations (W3C format)

GET /api/v1/manifests/{id}/canvas/{n}
→ Specific canvas with annotations

→ Your annotated corpus becomes a queryable dataset

For Machine Learning:

Arvest can export your annotated regions as:

  • COCO format — for object detection
  • CSV/JSON — for text classification
  • IIIF + W3C annotations — for ML pipelines that work natively with IIIF

→ Annotate once, reuse everywhere.

Consult docs.arvest.app/api for full API documentation

Tropy → structured metadata · Omeka/Nakala → persistent, IIIF-enabled storage · Arvest → collaborative annotation and machine-readable output

Summary & Key Takeaways

Conceptual foundations:

  • ✅ Use URIs instead of free text for people, places, concepts
  • ✅ Choose a metadata schema appropriate to your sources (DC, CIDOC-CRM)
  • ✅ Link to controlled vocabularies (GeoNames, Pactols, Openthéso)
  • ✅ Understand IIIF as the interoperability layer for images

Tools:

  • ✅ Tropy for the desktop: organization, description, annotation
  • ✅ Arvest for the web: IIIF import, collaboration, exposure
